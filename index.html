1. Edit Graph -- How Myers Thinks About Diff

The key insight in Myers' paper is that finding the diff between two files is actually the same problem as finding the shortest path through a grid. I didn't fully get this the first time I read it, but once it clicks it makes the whole algorithm obvious.

You build a grid with (m+1) columns and (n+1) rows where m = len(A) and n = len(B). Each point (i,j) just means "I've looked at i lines from A and j lines from B so far."

-> right: (i,j) -> (i+1, j ) we deleted A[i] cost = 1 ↓ down: (i,j) -> (i, j+1) we inserted B[j] cost = 1 ↘ diagonal: (i,j) -> (i+1, j+1) A[i] == B[j], free! cost = 0 (called a "snake")

So finding the minimal diff = finding the path from (0,0) to (m,n) that uses as few right/down moves as possible. The diagonal moves are free because matching lines don't cost anything. Edit distance D = number of non-diagonal moves.

Key formula:  D = |A| + |B| - 2.LCS(A,B)  ->  LCS = (|A|+|B|-D) / 2

This means the longer the common subsequence between the two files, the smaller D is -- which makes sense. If the files are identical, LCS = N and D = 0.

2. The V[k] Trick -- How Myers Gets O(N) Space

This was the part that took me longest to understand. Myers defines a diagonal number k = x - y. Every point on the same diagonal has the same k value. The clever part: instead of storing the whole grid, you only need to track the furthest-right x you've reached on each diagonal. That's V[k].

So instead of an O(mn) table, you just have a single array V of size 2N+1. The algorithm expands outward -- for d=0, d=1, d=2 ... -- and at each d it tries all odd or even diagonals (they alternate). The first time it reaches (m,n), D is found.

def myers_forward(A, B): m, n = len(A), len(B) V = {1: 0} # V[k] = furthest x we've reached on diagonal k trace = [] # need to save each step for the traceback later for d in range(0, m+n+1): trace.append(dict(V)) for k in range(-d, d+1, 2): # decide: did we come from above (insert) or left (delete)? if k == -d or (k != d and V[k-1] < V[k+1]): x = V[k+1] # came from above = insert else: x = V[k-1] + 1 # came from left = delete y = x - k # follow any free diagonal moves (matching lines) while x < m and y < n and A[x] == B[y]: x += 1 y += 1 V[k] = x if x >= m and y >= n: return backtrack(trace, A, B, d) def backtrack(trace, A, B, D): # walk backwards through the saved snapshots to rebuild the edit list ops = [] x, y = len(A), len(B) for d in range(D, 0, -1): V = trace[d] k = x - y prevK = k+1 if (k == -d or (k != d and V[k-1] < V[k+1])) else k-1 px = V[prevK] py = px - prevK while x > px and y > py: ops.insert(0, {'op': 'eq', 'a': x-1, 'b': y-1}) x -= 1; y -= 1 if x == px: ops.insert(0, {'op': 'ins', 'b': y-1}); y -= 1 else: ops.insert(0, {'op': 'del', 'a': x-1}); x -= 1 return ops

3. Why It's Actually O(ND)

Theorem 1 -- Time: Θ(ND)

At each value of d, we scan (2d+1) diagonals. The snake steps (free diagonal slides) can total at most m+n across the whole run, because each element can only be matched once. So total work = sum of diagonal scans + snake cost = Σ(2d+1) + O(N) = O(D^2) + O(N). Since D ≤ N this gives O(ND). The reason this matters: for a typical commit where D is small (say 10 changes in a 500-line file), it's basically O(N) in practice.

Theorem 2 -- Space: O(N)

At any point in the algorithm, we only need V[-d..d]. That's at most 2N+1 integers -- way less than an O(mn) grid. The traceback needs snapshots too but each snapshot is also O(N), and there are at most D snapshots. Myers also describes a divide-and-conquer version that reduces this further but I implemented the snapshot approach here since it's easier to visualize.

Theorem 3 -- Space-Optimized LCS: O(min(m,n))

For the LCS length itself (not traceback), you can get away with just two rows: prev[] and curr[]. You swap them each iteration and never look further back. This is "Variation 1" in the paper. It means for a 1000-line file you need 2000 integers instead of 1,000,000 -- about 500x less memory.

Here's the two-row implementation -- I tested this against the naive O(mn) version and they give the same results:

# two-row LCS -- only needs O(min(m,n)) space # swap A/B so B is always the shorter one def lcs_len(A, B): if len(A) < len(B): A, B = B, A m, n = len(A), len(B) prev = [0] * (n + 1) curr = [0] * (n + 1) for i in range(1, m + 1): for j in range(1, n + 1): if A[i-1] == B[j-1]: curr[j] = prev[j-1] + 1 else: curr[j] = max(prev[j], curr[j-1]) prev, curr = curr, [0] * (n + 1) # reuse curr next round return prev[n] # quick sanity check: # lcs_len(["a","b","c"], ["a","c"]) == 2 -- correct # # memory: N=1000 -> 2x1001 ints vs 1001x1001 = ~500x savings

4. Why the Old Approach Doesn't Scale

Before reading this paper I thought the standard LCS DP table was fine. But once you look at the numbers it's pretty clear why Git couldn't use it:

# Real-world numbers Classical LCS on two 1000-line files: time: O(1000 * 1000) = 1,000,000 operations memory: 1,000,000 integers ~ 4MB just for the table Myers on same files with D=20 changes: time: O(1000 * 20) = 20,000 operations -- 50x faster memory: ~2,000 integers -- 500x less # On a real repo (linux kernel, ~70k files): # Classical would need gigabytes just for the DP tables # Myers stays fast because most commits only touch a few lines

The reason Myers works so well in practice is that D (the number of actual changes) is usually tiny compared to N. A 500-line file with a 3-line bugfix has D=3. Myers essentially does O(N) work in that case. The worst case is D=N (completely different files) which gives O(N^2), but that almost never happens in real version control.

Git actually uses a variant called "histogram diff" by default now, which handles repeated lines better. But Myers is still the baseline that everything else is compared against, and it's what's implemented here.
